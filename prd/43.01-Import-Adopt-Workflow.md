# PRD: Adoption Workflow & Import

**Status:** Part 1 — Ready for Implementation | Part 2 — Future  
**Created:** 2026-02-11  
**Version:** 0.21.0+

## Context

Adoption is the process of bringing **existing dbt Cloud infrastructure** under Terraform management. This applies in two scenarios:

1. **Migration Adoption (Part 1)** — Within the Migration workflow, users adopt target resources into Terraform management. This covers two sub-scenarios:
   - **Source-matched adoption:** A source resource is matched to an existing target resource. The target's IDs and configuration are used so Terraform manages the existing resource without recreating it.
   - **Target-only adoption:** A target resource exists with no corresponding source resource. The user can still choose to bring it under Terraform management (e.g., the target account has projects D and E that don't exist in the source).

2. **Import & Adopt (Part 2)** — A future standalone workflow that works purely against a single target account (no source). Users select which target resources to bring under Terraform management.

Part 1 is the immediate focus. All components built for Part 1 should be designed for reuse by Part 2. The target-only adoption sub-scenario in Part 1 directly exercises the same code paths that Part 2 will use exclusively.

#### 1a. Match Scope Visibility Filter

The Match grid currently shows state-only rows (resources in TF state with no source counterpart) regardless of source scope. With target-only adoption (section 1b), target-only rows will also appear. A scope visibility filter lets users focus the grid on what they selected in the Scope step.

**This is a pure visibility filter.** It does NOT affect intent computation, protection logic, import block generation, YAML output, or any Terraform operations. Hidden rows retain their current action/state — they are just not displayed.

- **Filter OFF (full view, default):** All rows visible — source-scoped items, state-only items, target-only items.
- **Filter ON (scoped view):** Only rows connected to source-scoped items are visible. State-only and target-only rows that have no relationship to any scoped source item are hidden.

A row is "connected to a scoped source" if:
- It IS a source item in the scope selection (matched or unmatched)
- It is a target/state resource that is matched to a scoped source item

| ID | Story | Acceptance Criteria |
|----|-------|---------------------|
| US-SV-01 | As a migration operator, I want a "Show Scoped Only" toggle in the Match grid toolbar | Toggle appears alongside the existing type filter; default is OFF (full view) |
| US-SV-02 | As a migration operator, I want the scoped view to show only source-scoped rows and their matches | When toggle is ON, only rows whose source item is in the SelectionManager's selected_ids appear; their matched target data is included |
| US-SV-03 | As a migration operator, I want state-only rows hidden when scoped view is active | State-only rows (`is_state_only: True`) are hidden when toggle is ON, since they have no scoped source counterpart |

#### 1c. Target-Only Adoption Preference

Not all migration operators want to be presented with target-only resources. A two-level preference controls the default visibility:

- **Project-level default** — persists across sessions (stored in project config / `.env`). Once set, the Match page respects it without re-asking.
- **Per-session override** — the "Show Target-Only Resources" toggle in the grid toolbar overrides the project default for the current session.

A smart first-run dialog appears **only when unmatched target resources are detected** (target resources that don't match any source resource and aren't already in TF state). If all target resources are either matched or in state, the dialog does not appear.

| US-TP-06 | As a migration operator, I want the dialog to NOT appear when all target resources are already matched or in state | If no unmatched target-only resources exist, the dialog is skipped entirely regardless of preference state |

### Architectural Decisions

| ID | Decision | Rationale |
|----|----------|-----------|
| AD-1 | **Project granularity: Hybrid approach** — Grid organized by project for viewing; "Select Whole Project" dialog with child counts and customization; cherry-pick individual resources | Balances ease of use (bulk project adoption) with fine-grained control (individual resource selection) |
| AD-2 | **Dependency handling: Dialog assist, non-blocking** — Cascade-style dialog suggests adopting dependencies; user can skip | Matches 31.02 protection cascade pattern; non-blocking respects user autonomy while warning about risks |
| AD-3 | **Intent format: Reuse target-intent.json** — Add `adopt` and `ignore` dispositions alongside existing `match`/`create` | Single intent file simplifies state management; disposition types are workflow-aware |
| AD-4 | **Import block format: TF 1.5+ import {} blocks** — Declarative blocks in config files | Declarative approach is plannable/reviewable; TF 1.5 is 3+ years old; legacy CLI fallback already exists in `terraform_import.py` |
| AD-5 | **Target-only display: Same grid, visually distinguished** — Target-only resources appear in the Match grid alongside source-matched rows with a "Target Only" badge, empty Source column, and filter toggle | Dependencies cross the boundary (target-only job may depend on source-matched project); single grid enables cascade dialogs; filter toggle lets users focus; directly exercises Part 2 code paths |
| AD-6 | **Target-only preference: Two-level opt-in** — Project-level default (persists across sessions) + per-session toggle override; smart first-run dialog only when unmatched target resources detected | Respects users who just want to migrate without noise; smart trigger avoids unnecessary dialogs; project-level persistence prevents repeated asking; source-to-target matching always available regardless |
| AD-7 | **Match scope visibility filter: Pure UI filter** — "Show Scoped Only" toggle hides state-only and target-only rows not connected to source scope; does not affect intent, protection, imports, or any TF logic | Keeps grid focused for users migrating specific projects; composable with type filter and target-only preference; hidden rows retain their state |

## Part 2: Import & Adopt Workflow (Future)

### Vision

A standalone workflow that works purely against a single target dbt Cloud account — no source account needed. Users fetch their target account, review what exists, select which resources to bring under Terraform management, and deploy.

### Key Differences from Migration

| Aspect | Migration Workflow | Import & Adopt |
|--------|-------------------|----------------|
| Source account | Required | Not needed |
| Driver | Source items drive grid; target matches | Target items drive grid |
| Default action | Match (adopt existing) or Create New | Adopt (import to TF) or Ignore |
| YAML generation | Source YAML normalized, then target overrides applied | Target fetch directly normalized to config via `normalize_target_fetch()` |
| Import blocks | Generated for matched/adopted resources | Generated for all adopted resources |
| State-only items | Inherited from prior apply | Not applicable (fresh start) |

### Reusability Map: Part 1 Components in Part 2

Part 1's target-only adoption code path (US-TO-01 through US-TO-10) is the **direct precursor** to Part 2. When the Import & Adopt workflow runs, _every_ resource is "target-only" — there is no source. The same grid rendering, YAML generation, import block creation, and dependency cascade logic applies.

| Part 1 Component | Part 2 Reuse | Adaptation Needed |
|-------------------|-------------|-------------------|
| `build_grid_data()` in `match_grid.py` | Drives the `/adopt` page grid | Part 2 = all rows are "target-only" mode from Part 1; remove Source column entirely; default action = "adopt" instead of "ignore" |
| Target-only YAML generation via `normalize_target_fetch()` | Generates all YAML in Part 2 | No changes — Part 1 target-only adoption already exercises this exact code path |
| `terraform_import.py` | Generates import blocks for all adopted resources | No changes — already workflow-agnostic |
| `adoption_yaml_updater.py` | Applies target values to YAML | No changes — already works on target data |
| `ProtectionIntentManager` in `protection_intent.py` | Tracks protection for adopted resources | No changes — already workflow-agnostic |
| Cascade dependency dialogs (Part 1 US-DH-01 through US-DH-03) | Reused for dependency-assist in adopt grid | No changes — same dialog, same hierarchy_index lookups |
| `hierarchy_index.py` | Dependency resolution for adopt cascades | No changes |
| `target-intent.json` with `adopt`/`ignore` dispositions (AD-3) | Intent file for the adopt workflow | Same format; workflow type distinguishes migration vs. standalone |
| "Target Only" badge and filter toggle (Part 1 AD-5) | Not needed in Part 2 — all rows are target resources | Toggle removed; badge removed; grid is purely target-driven |

### Three Resource Categories

1. **Target resources** — exist in the target dbt Cloud account. User decides: **Adopt** (bring under TF management) or **Ignore** (leave unmanaged).

2. **State resources** (Day 2 scenario) — resources already in `terraform.tfstate` from a prior partial adoption. Shown as "Already Managed" and auto-included.

3. **Global resources** — account-level resources (groups, service tokens, etc.) follow patterns from [42.01 Part 2](42.01-Global-Resources-Target-Intent.md) (future work — dependency acknowledged).

### Proposed Steps

```
1. Fetch Target          — reuse existing /fetch_target page (no source needed)
2. Explore Target        — reuse existing /explore_target page
3. Select & Adopt        — NEW page: /adopt (target-driven grid, reuses match_grid patterns)
4. Configure             — reuse existing /configure page
5. Target Credentials    — reuse existing /target_credentials page
6. Deploy                — reuse existing /deploy page (with import block generation)
```

### The `/adopt` Page (Future)

Reuses Part 1's `build_grid_data()` and grid patterns with target-only configuration:

#### Grid Columns

| Column | Type | Notes |
|--------|------|-------|

| Name | Text | Target resource name |
| Target ID | Number | dbt Cloud resource ID |
