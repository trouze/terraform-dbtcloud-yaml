# PRD: Global Resources in Target Intent

**Status:** Part 1 — Ready for Implementation | Part 2 — Future  
**Created:** 2026-02-10  
**Version:** 0.19.0+  

## Context

Global resources (groups, service_tokens, notifications, privatelink_endpoints, webhooks) currently leak into the generated Terraform config from the source YAML without explicit user intent. v0.19.0 added `included_globals` filtering to `compute_target_intent()` which defaults to only `connections` and `repositories`. This PRD defines how to surface that control to users.

**Connections and repositories are out of scope** -- they flow implicitly with project matching and should not be changed.

---

## Part 1: Global Resources Checkbox Card (Immediate)

### Goal

Give users explicit, visible control over which non-project global sections are included in the target intent, without adding a new workflow step or complicating the Match page.

### Design

Add a **"Global Resources"** card to the **Configure Migration** page (`importer/web/pages/configure.py`), in the left column between Migration Summary and Mapping Summary.

#### Card Layout

```
Global Resources
─────────────────────────────────────────
Connections and repositories are managed automatically
via project matching. Other global resources require
explicit opt-in.

  [ ] Groups (3 in source)
  [ ] Service Tokens (2 in source)
  [ ] Notifications (skipped — not yet supported in TF)
  [ ] PrivateLink Endpoints (read-only)

  ⚠ 2 global types already in TF state: Groups (3),
    Service Tokens (1). These will be auto-retained
    to prevent destruction.
```

- Checkboxes unchecked by default (matching current `included_globals` default)
- Counts pulled from source YAML (`state.map.yaml_file` or output_config globals)
- Warning banner when TF state contains globals that are NOT checked -- auto-retain safety net
- Notifications and PrivateLink shown as disabled/informational (TF doesn't manage them today)

#### State Changes

Add to `MapState` in `importer/web/state.py` (near existing `include_groups` at line 367):

```python
# Replace the existing include_* booleans with a single set
selected_globals: set[str] = field(default_factory=lambda: set())
```

Or reuse the existing `include_groups`, `include_service_tokens`, etc. booleans (lines 367-371) -- they already exist but are only used for source scope, not target intent. The simplest path is to **reuse them** and wire them into `compute_target_intent()`.

#### Wiring

1. **Configure page** reads/writes `state.map.include_groups`, `state.map.include_service_tokens`, etc.
2. **Match page** (`importer/web/pages/match.py` line 314): build `included_globals` set from state flags and pass to `compute_target_intent()`
3. **Deploy page** (`importer/web/pages/deploy.py` line 1521): same -- build and pass `included_globals`
4. Both call sites already have access to `state`

```python
# Helper to build included_globals from state flags
def _build_included_globals(state: AppState) -> set[str]:
    result = {"connections", "repositories"}  # always included
    if state.map.include_groups:
        result.add("groups")
    if state.map.include_service_tokens:
        result.add("service_tokens")
    # notifications/privatelink/webhooks: future
    return result
```

#### Safety Net: Auto-Retain from TF State

Add a helper to detect which global sections exist in TF state:

```python
def get_tf_state_global_sections(tfstate_path: Path) -> set[str]:
    """Return set of global section keys that have resources in TF state."""
```

This scans state for `dbtcloud_group` -> `"groups"`, `dbtcloud_service_token` -> `"service_tokens"`, etc. using the existing `TF_TYPE_TO_CODE` mapping in `importer/web/utils/terraform_state_reader.py`.

At intent computation time, auto-add any TF-state-present globals to `included_globals` regardless of checkbox state, and log a warning:

```
WARN: Auto-retaining globals.groups (in TF state) despite not being selected
```

This prevents accidental destruction.

## Part 2: Full Global Resource Matching (Medium-Term)

### Vision

Groups and service tokens get the same match/adopt/retain treatment as projects. Users can see "Source Group X matches Target Group Y" and choose to adopt, create new, or ignore.

### User Stories

- **US-1**: As a migration operator, I want to see which groups exist on both source and target so I can adopt existing groups instead of creating duplicates.
- **US-2**: As a migration operator, I want to adopt a target service token so Terraform manages it without recreating it.
- **US-3**: As a migration operator, I want to protect specific groups from modification, just like I protect projects.
- **US-4**: As a migration operator, I want to see a "Global Resources" tab/section on the Match page showing group and token matches alongside project matches.
- **US-5**: As a migration operator, I want the system to auto-match groups by name (e.g., "Owner" matches "Owner") and flag conflicts.
- **US-6**: As a migration operator, I want to remove a group from management without Terraform destroying it (safe unmanage).

### Architecture Decisions Needed

**AD-1: Disposition model for globals**

- Do globals get `ResourceDisposition` entries like projects? Or a separate `GlobalDisposition`?
- If using `ResourceDisposition`, the `resource_type` field (`"PRJ"`, `"GRP"`, `"TOK"`) already supports this
- Implication: dispositions dict keys would need namespacing (e.g., `"GRP:owner"`, `"TOK:deploy_token"`)

**AD-2: Matching algorithm for globals**

- Projects match by key/name/ID. Groups match by... name? SSO mapping? ID?
- Service tokens: match by name? Purpose?
- What about partial matches? (Source "Owner" vs Target "Owners")

**AD-3: Protection for globals**

- Currently protection only applies to projects (protected_projects in TF)
- The TF module has no `protected_groups` or `protected_service_tokens`
- Would need module-level changes to support lifecycle { prevent_destroy } on globals

**AD-4: UI placement**

- Option A: Separate "Global Resources" tab on Match page
- Option B: Dedicated "Select Target" step (new page between Explore Target and Match)
- Option C: Inline sections on Match page (projects section + globals section)
- Recommendation from Part 1 discussion: Start with Option A (tab) and graduate to Option B if complexity warrants it

**AD-5: State management for global dispositions**

- Stored in `target-intent.json` alongside project dispositions?
- Or separate `globals-intent.json`?
- Recommendation: Same file, namespaced keys (`GRP:owner`, `TOK:deploy`)

### Open Questions

- **OQ-1**: How do we handle group permissions that reference project IDs? Group permissions have `project_id` which maps to project keys. If a group references a project we're not managing, what happens?
- **OQ-2**: Service tokens have secrets. Can we adopt a service token without knowing its secret? The TF provider may need special import handling.
- **OQ-3**: Should we support "create on target but don't match" for globals? (e.g., user wants a new "Migration Admin" group that doesn't exist on source)
- **OQ-4**: What's the interaction between global selection and the normalizer's `is_resource_included("groups")` config? Today the normalizer has its own include/exclude logic. Do we need to reconcile?
- **OQ-5**: For groups with project-scoped permissions, should group selection be dependent on project selection? (e.g., if you select projects A and B, auto-include groups that have permissions on A or B?)
- **OQ-6**: How do we handle the case where source has 20 groups and target has 20 groups but they don't map 1:1? Full match grid? Or just name-based auto-match with manual override?

### Test Specifications (Future)

- **T-1**: Group auto-match by name: source "Owner" matches target "Owner"
- **T-2**: Group with no target match shows as "create new"
- **T-3**: Adopted group retains target ID in TF state

- **T-5**: Group with project-scoped permissions updates project_id references after project adoption
- **T-6**: Service token adoption preserves token value (or handles import correctly)
- **T-7**: Removing a global from selection when it's in TF state triggers warning, not destruction
- **T-8**: Round-trip: global dispositions serialize to/from target-intent.json correctly
- **T-9**: Match page tab shows globals grid with correct counts and actions
