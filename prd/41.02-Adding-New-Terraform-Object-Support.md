# PRD: Adding New Terraform Object Support — One-Stop Guide

## Overview

This document is the **one-stop checklist and guide** for planning and building support for **new Terraform / dbt Cloud resource types** in the terraform-dbtcloud-yaml importer and web UI. Use it whenever you add a new resource (e.g. a new `dbtcloud_*` resource) so that fetch, YAML, protection, Destroy/Match pages, and tests are all covered and nothing is missed.

**When to use this guide:**

- Before writing a resource-specific PRD (e.g. 41.01 Extended Attributes): use the checklist to scope the work.
- When implementing a new resource type: work through each section and touchpoint.
- When reviewing a PR for a new resource: verify every relevant subsection is addressed.

**Reference implementation:** [41.01-Extended-Attributes-Support.md](41.01-Extended-Attributes-Support.md) and the lessons in [.cursor/plans/extended_attributes_next_steps_29eb9c18.plan.md](../.cursor/plans/extended_attributes_next_steps_29eb9c18.plan.md).

## 1. Data Model & Fetch

- [ ] **Model:** Add or extend Pydantic models in `importer/models.py` (e.g. new resource model, parent model fields like `extended_attributes_key`).
- [ ] **Fetcher:** Implement fetch in `importer/fetcher.py` (paginated if needed, filter deleted if applicable).
- [ ] **Keys:** Define a stable composite key (e.g. `project_key_ext_key`) for the resource and use it consistently.
- [ ] **Linking:** If this resource is referenced by another (e.g. environment → extended attributes), populate the linking field(s) after fetch.
- [ ] **Indentation audit:** After editing `fetcher.py`, verify indentation around `project_done_flags` / `if progress:` / `with ThreadPoolExecutor` — see Section 15.
- [ ] **Fetch completion counts:** In `fetch_source.py`, add the new type to `resource_counts` dict (line ~731) and to the terminal summary lines (line ~770). Without this, the type won't appear in the post-fetch summary even though it was fetched.

**Reference:** 41.01 Phase 1 (US-1.x).

## 2. YAML Schema & Serialization

- [ ] **Schema:** Update `schemas/v2.json`: add `$defs` for the resource, add to parent objects (e.g. `projects[].extended_attributes`), add `protected` if the resource supports protection.
- [ ] **Converter:** Update `importer/yaml_converter.py` to read/write the new structure.
- [ ] **Normalizer:** Update `importer/normalizer/core.py` so the resource is normalized and keys are consistent.
- [ ] **Import mapping:** Update `importer_mapping.yml` if the resource is part of import/mapping flows.

**Reference:** 41.01 Phase 2 (US-2.x).

## 3. Protection Workflow — Core Maps

Protection touches several maps. **All** must include the new type where applicable.

### 3.1 RESOURCE_TYPE_MAP (`importer/web/utils/protection_manager.py`)

Used for: **extract** (YAML → protected set), **detect** (protection changes), **get_resource_addresses** (basic address generation).

- [ ] Add entry: `"NEWTYPE": ("dbtcloud_new_resource", "unprotected_block_name", "protected_block_name")`.
- [ ] Ensure composite key (e.g. `project_key_ext_key`) is used in `extract_protected_resources` / `build_protection_map` so the new type is detected from YAML and state.

### 3.2 EXTENDED_RESOURCE_TYPE_MAP (`importer/web/utils/protection_manager.py`)

Used for: **mismatch/repair**, **SingleResourceProtectionInfo** (`state_address_prefix`, `expected_address_prefix`), and any logic that resolves addresses for "extended" (non-JOB/ENV/PRJ/REP/PREP) types.

- [ ] Add entry with same shape: `(tf_resource_type, unprotected_block, protected_block)`.
- [ ] If omitted, the new type will resolve to `"unknown"` in address generation and mismatch repair.

### 3.3 Hierarchy index (`importer/web/components/hierarchy_index.py`)

Used for: **cascade protect/unprotect**, **Select Parents / Select Children** in Scope and Match.

- [ ] **ENTITY_PARENT_TYPES:** Add `"NEWTYPE": ["PRJ"]` (or correct parent type list).
- [ ] **TYPE_DEPTH:** Add depth for sorting (e.g. project-scoped = 2).
- [ ] **TYPE_SORT_ORDER:** Add value so the type appears in the right order in UI.
- [ ] **Reporter:** Ensure report/source items include this resource with `element_type_code: "NEWTYPE"` and correct parent reference so the hierarchy index can build the graph.

### 3.4 Secondary type maps in protection_manager.py

These are **easily missed** because they're scattered across different functions, not grouped with the primary maps:

- [ ] **TYPE_LABELS** (line ~657): Display label for cascade UI. Missing = raw code shown.
- [ ] **`format_protection_warnings()` type_names** (line ~601): Display label for warning messages. Missing = raw code shown.
- [ ] **`generate_repair_moved_blocks()` type_names** (line ~1266): Used in moved-block repair. Missing = raw code in output.
- [ ] **`format_mismatches_for_display()` type_names** (line ~1449): Mismatch display. Missing = raw code shown.
- [ ] **`detect_protection_mismatches()`** (line ~1152): Currently only maps `dbtcloud_project`, `dbtcloud_repository`, `dbtcloud_project_repository`. Add the new TF type → type code mapping here too.

> **EXTATTR gap found Jan 2026:** All five of these secondary maps were missing EXTATTR. The primary maps (RESOURCE_TYPE_MAP, EXTENDED_RESOURCE_TYPE_MAP) were correct, but these secondary maps silently fell back to raw type codes.

### 3.5 Linked resources (if applicable)

If the new resource has a **bidirectional** relationship (e.g. ENV ↔ Extended Attributes by reference), not just parent→child:

- [ ] **get_resources_to_protect:** When protecting A, add linked B(s); when protecting B, add linked A(s). Implement in `protection_manager.py`.
- [ ] **get_resources_to_unprotect:** When unprotecting A, include linked B(s) in "protected children"; same for B.
- [ ] **Select source (Scope/Match):** Add `get_linked_entities(mapping_id)` in `hierarchy_index.py` and use it when building "Select Parents" / "Select Children" so linked entities are selected together. Update `scope.py` and `mapping.py` to include linked entities in the selection set.

**Reference:** Extended Attributes plan — "Linked-resource rule: ENV ↔ EXTATTR".

## 15. Fetch Error Masking — Critical Anti-Pattern

> **Lesson learned (EXTATTR rollout, Jan 2026):** An `IndentationError` in `fetcher.py` caused the `from importer.fetcher import ...` to fail inside `fetch_source.py`. Because `FetchCancelledException` was pre-declared as `Exception` (a safety fallback for missing imports), the `except FetchCancelledException:` handler caught the `IndentationError` and displayed **"FETCH CANCELLED"** — completely masking the real syntax error. No debug logs were produced because the code never reached them.

### Guard against this

- [ ] **Always syntax-check `fetcher.py`** (and any module touched during a change) before committing. Run `python -c "import importer.fetcher"` from the repo root. **Add this as a CI gate.**
- [ ] **Never rely solely on UI error messages** when debugging fetch failures. The "FETCH CANCELLED" path can mask *any* exception if the import fails.
- [ ] **When debugging fetch issues:** Add instrumentation at the **very top** of the `try:` block (before imports) and in **each** `except` handler. The debug log should capture `exc_type`, `exc_str`, and whether `FetchCancelledException is Exception` (which reveals the fallback was used).
- [ ] **Indentation discipline in `fetcher.py`:** The fetcher uses nested functions (`_check_cancelled`, `_fetch_with_retry`, etc.) and parallel blocks inside `ThreadPoolExecutor`. An extra 4 spaces of indentation turns valid code into a syntax error. Always review indentation after edits to `fetcher.py`, especially around the `project_done_flags` / `if progress:` / `with ThreadPoolExecutor` boundary.
- [ ] **Cancel event race:** Clear `cancel_event` inside the `do_fetch` thread function (not in the caller before `asyncio.to_thread`), so any phantom set() between scheduling and thread start is ignored.

### Recommended debug instrumentation pattern

When the fetch page shows unexpected behavior, add this at the top of `_run_fetch`'s `try:` block:

```python
_DL = "/tmp/terraform_dbtcloud_fetch_debug.log"
try:
    open(_DL, "a").write(json.dumps({
        "location": "try_start", "timestamp": time.time()
    }) + "\n")

    from importer.fetcher import fetch_account_snapshot, FetchCancelledException

    open(_DL, "a").write(json.dumps({
        "location": "imports_done",
        "FCE_is_Exception": FetchCancelledException is Exception
    }) + "\n")
    ...
except FetchCancelledException as e:
    open(_DL, "a").write(json.dumps({
        "location": "except_FCE",
        "exc_type": type(e).__name__,
        "exc_str": str(e),
        "FCE_is_Exception": FetchCancelledException is Exception
    }) + "\n")
except Exception as e:
    import traceback
    open(_DL, "a").write(json.dumps({
        "location": "except_generic",
        "exc_type": type(e).__name__,
        "exc_str": str(e),
        "traceback": traceback.format_exc()[-2000:]
    }) + "\n")
```

**Key signal:** If `FCE_is_Exception` is `True` in the `except_FCE` handler, the real `FetchCancelledException` was never imported — the exception is being misclassified.

**Always use `/tmp/` as the log path** so it succeeds regardless of workspace permissions.

## 16. Debugging Standards

These standards apply to any debugging session, not just fetch issues.

### 14.1 Log location hierarchy

1. `/tmp/<descriptive_name>.log` — always works, use as primary
2. `<repo>/.cursor/debug.log` — convenient but may fail on permissions
3. `<repo>/dev_support/debug.log` — fallback inside repo

Write to the first path that succeeds. Use `json.dumps()` with a newline for structured, parseable logs.

### 14.2 Debug log format

Every log entry should be a single JSON line containing at minimum:

| Field | Purpose |
|-------|---------|
| `location` | Code path identifier, e.g. `"fetcher:_check_cancelled"` |
| `timestamp` | `time.time()` — for ordering and latency measurement |
| `message` | Human-readable description of what happened |

For exception handlers, also include:

| Field | Purpose |
|-------|---------|
| `exc_type` | `type(e).__name__` |
| `exc_str` | `str(e)` (truncate if needed) |
| `traceback` | `traceback.format_exc()[-2000:]` |

### 14.3 Instrumentation lifecycle

1. **Add instrumentation** with `# region agent log` / `# endregion agent log` markers so they can be found and removed.
2. **Verify instrumentation fires** — if no logs appear, the code path is not being reached (the bug is earlier).
3. **Do NOT remove instrumentation** until the fix is verified with post-fix logs.
4. **Clean up** all `# region agent log` blocks before merging to main.

### 14.4 Threading and asyncio pitfalls

- `threading.Event.set()` / `.clear()` between the caller and `asyncio.to_thread(do_fetch)` is a race. Clear inside the thread.
- `asyncio.CancelledError` can be raised by the event loop, not just by user cancellation. Always log which path raised it.
- Pre-declaring fallback exception classes (`FetchCancelledException = Exception`) masks import failures. If you use this pattern, always log `FetchCancelledException is Exception` to detect the fallback being active.

## Dependency Order (Implementation)

Rough order that avoids rework:

1. **Data model & fetch** → Reporter/source items (so hierarchy has data).
2. **Protection maps & hierarchy_index** (RESOURCE_TYPE_MAP, EXTENDED_RESOURCE_TYPE_MAP, ENTITY_PARENT_TYPES, TYPE_DEPTH, TYPE_SORT_ORDER, linked resources if any).
3. **Terraform state reader** and **Destroy page** address mapping and UI.
4. **Match/Scope** (grid, protection column, select parents/children + linked).
5. **Fixtures** (YAML, state, protection-intent, report items for all types).
6. **Unit tests** (protection_manager, extract, detect).
7. **Playwright E2E** (Destroy, then Match).
8. **CI gates** (import smoke test for fetcher, abbreviation consistency check).
