# Release Notes: v0.5.0

**Release Date:** 2025-12-20  
**Type:** Minor Release (New Feature)  
**Status:** ✅ Ready for Use

---

## Overview

Version 0.5.0 introduces **interactive credential saving** to streamline the workflow for users who frequently use the importer. This release adds the ability to save credentials entered during interactive mode to `.env` files, eliminating the need to re-enter credentials in future sessions.

### Key Features

- **Save Credentials to .env**: Persist credentials entered interactively for future sessions
- **Connection Credential Prompting**: Interactive prompts for connection provider_config during normalization
- **Enhanced E2E Workflow**: Save connection credentials after provider_config setup
- **Security Features**: Automatic file permissions, gitignore warnings, and overwrite protection

---

## What's New

### 1. Interactive Credential Saving

**Problem Solved:**  
Users had to manually create `.env` files or re-enter credentials every time they used interactive mode, which was tedious for frequent users.

**Solution:**  
After entering credentials interactively, users are now prompted to save them to `.env` file for future sessions.

**User Experience:**
```
? Save credentials to .env file for future sessions? (Y/n)
```

**If Yes:**
- Credentials are written to `.env` file at repo root
- File permissions automatically set to 600 (read/write owner only)
- If `.env` already exists, user can choose append or overwrite
- If `.env` is not in `.gitignore`, user is warned and can add it automatically

**Technical Changes:**
- **File:** `importer/interactive.py`
- **New Function:** `save_credentials_to_env(credentials, connection_configs=None)`
- **Helper Functions:**
  - `_get_env_file_path()`: Returns repo root `.env` path
  - `_read_existing_env()`: Parses existing .env into dict
  - `_write_env_file()`: Writes credentials with proper formatting
  - `_format_env_value()`: Handles value escaping (quotes, spaces)
  - `_check_gitignore()`: Checks if .env is in .gitignore

**Integration:**
- Integrated into `run_fetch_interactive()` after credential entry
- Only prompts if credentials were entered (not already in environment)

**Example .env Output:**
```bash
# dbt Cloud Importer Credentials
# Generated by interactive mode
# DO NOT commit this file to version control

# Source Account Credentials (for fetch)
DBT_SOURCE_HOST=https://cloud.getdbt.com
DBT_SOURCE_ACCOUNT_ID=12345
DBT_SOURCE_API_TOKEN=your_token_here
```

### 2. Connection Credential Prompting

**Problem Solved:**  
Users had to manually edit YAML files to add connection `provider_config` after normalization, which was error-prone and time-consuming.

**Solution:**  
After normalization completes, if connections are missing `provider_config`, users can configure them interactively with type-specific prompts.

**User Experience:**
```
Some connections need provider_config. Configure now? (Y/n)
```

**If Yes:**
- For each connection missing `provider_config`, prompts for type-specific fields:
  - **Snowflake:** account, database, warehouse, role (optional)
  - **Databricks:** host, http_path, catalog (optional)
  - **BigQuery:** project_id, dataset, location
  - **Redshift:** hostname, port, dbname
  - **PostgreSQL:** hostname, port, dbname
- Updates YAML file with provider_config
- Offers to save connection credentials to `.env`

**Technical Changes:**
- **File:** `importer/interactive.py`
- **New Function:** `prompt_connection_credentials(yaml_file: Path) -> dict`
- **Integration:** Added to `run_normalize_interactive()` after normalization completes

**Example Flow:**
```
Connection: Snowflake Production
Key: snowflake_prod, Type: snowflake

? Snowflake Account: abc12345
? Database: prod_db
? Warehouse: compute_wh
? Role (optional): transform_role

✓ Updated normalized YAML with provider_config
? Save connection credentials to .env file? (Y/n)
```

### 3. E2E Test Script Enhancement

**Problem Solved:**  
After adding provider_config in E2E test workflow, users had to manually extract and save connection credentials if they wanted to reuse them.

**Solution:**  
Added option to save connection credentials to `.env` after provider_config is added (either dummy or manual).

**User Experience:**
```
Save connection credentials to .env file for future use? (y/n) [n]:
```

**If Yes:**
- Extracts `provider_config` from YAML file
- Writes to `.env` in standardized format
- Groups credentials by connection with helpful comments

**Technical Changes:**
- **File:** `test/run_e2e_test.sh`
- **New Function:** `save_connection_credentials_to_env(yaml_file)`
- **Integration:** Added to `configure_provider_configs()` after provider_config is added

**Example .env Output:**
```bash
# Connection Provider Configs (from E2E test)

# Connection: snowflake_prod
DBT_CONNECTION_SNOWFLAKE_PROD_ACCOUNT=abc12345
DBT_CONNECTION_SNOWFLAKE_PROD_DATABASE=prod_db
DBT_CONNECTION_SNOWFLAKE_PROD_WAREHOUSE=compute_wh
DBT_CONNECTION_SNOWFLAKE_PROD_ROLE=transform_role

# Connection: databricks_dev
DBT_CONNECTION_DATABRICKS_DEV_HOST=workspace.cloud.databricks.com
DBT_CONNECTION_DATABRICKS_DEV_HTTP_PATH=/sql/1.0/warehouses/abc123
DBT_CONNECTION_DATABRICKS_DEV_CATALOG=main
```

## Security Features

### File Permissions
- `.env` files are automatically set to permissions 600 (read/write owner only)
- Prevents accidental credential exposure

### Gitignore Protection
- Checks if `.env` is in `.gitignore` before saving
- Warns user if not present
- Offers to add `.env` to `.gitignore` automatically

### Overwrite Protection
- Detects existing credentials in `.env`
- Prompts before overwriting existing values
- Option to append only new credentials

### Value Escaping
- Properly escapes values containing spaces, quotes, or special characters
- Wraps values in quotes when necessary

## User Workflows

### New User (First Time)
1. Run `python -m importer fetch --interactive`
2. Enter credentials when prompted
3. Choose to save credentials to `.env`
4. Next time: Credentials loaded automatically from `.env`

### Existing User (Has .env)
1. Run `python -m importer fetch --interactive`
2. Credentials loaded from `.env` automatically
3. No prompts needed

### Normalize with Connection Config
1. Run `python -m importer normalize --interactive`
2. After normalization, prompted to configure connections
3. Enter connection credentials interactively
4. Choose to save to `.env`
5. Connection credentials available for future use

## Backwards Compatibility

**Fully Backwards Compatible:**
- Existing `.env` files continue to work
- Manual credential entry still works (just don't save)
- No breaking changes to API or CLI

**Optional Feature:**
- Users can choose not to save credentials
- Existing workflows unchanged

## Known Limitations

- **Connection credentials in .env:** Currently stored as flat environment variables. Future enhancement could support structured storage or secret management integration.
- **Target account credentials:** Not yet prompted for saving (DBTCLOUD_*). Can be added manually to `.env` if needed.
- **Credential encryption:** `.env` files are stored in plaintext. Users should ensure proper file permissions and gitignore configuration.
