# Release Notes: v0.6.0

**Release Date:** 2025-12-20  
**Version:** 0.6.0 (Minor Release)  
**Type:** Feature Enhancement

---

## Summary

This minor release enhances the E2E test infrastructure with improved test mode functionality. Test mode now uses real connection data from `.env` files instead of dummy test values, and includes a new `--test-destroy` flag for cleaning up test resources. Additionally, a bug fix ensures test mode correctly targets resources for planning and applying.

---

## ✨ New Features

### Test Mode Destroy Flag (`--test-destroy`)

**Purpose:** Enable cleanup of test resources created during E2E testing.

**Features:**
- Targets 1-2 resources (connections preferred, groups fallback) using the same logic as `--test-plan` and `--test-apply`
- Supports standalone destroy mode (skips fetch/normalize when only destroy is requested)
- Includes 10-second warning before destroy execution for safety
- Uses `terraform destroy -auto-approve -target=...` for targeted resource removal

**Usage:**
```bash
# Destroy test resources after apply
./run_e2e_test.sh --test-apply --test-destroy

# Destroy test resources standalone (requires existing Terraform state)
./run_e2e_test.sh --test-destroy
```

**Benefits:**
- Enables iterative testing without manual resource cleanup
- Reduces risk of leaving test resources in target accounts
- Provides consistent cleanup workflow matching test creation workflow

## 🔄 Changed Behavior

### Test Mode Uses Real Connection Data

**Previous Behavior:** Test mode created connections with dummy test data (e.g., "test-host.cloud.databricks.com", "test-account", "TEST_DB")

**New Behavior:** Test mode reads connection configuration from `provider_config` field in YAML (populated from `.env` files), ensuring real connection credentials are used.

**Impact:**
- Test resources are created with actual configuration values
- More accurate testing of real-world scenarios
- Connections created in test mode match production configuration

**Technical Details:**
- Updated `modules/projects_v2/globals.tf` to prioritize `provider_config` over `details` for connection configuration
- Fallback order: `provider_config` → `details` → defaults
- Updated connection types: Databricks, Snowflake, BigQuery, Postgres, Redshift

**Example:**
```yaml
connections:
  - key: snowflake_prod
    name: Snowflake Production
    type: snowflake
    details:
      adapter_version: snowflake_v0
    provider_config:  # ← Now read first
      account: fka50167
      database: analytics
      warehouse: transforming
```

## 🐛 Bug Fixes

### Test Mode Resource Targeting

**Problem:** Test mode (`--test-plan`, `--test-apply`) was not correctly targeting resources. Terraform reported "No changes" even when resources didn't exist in the target account.

**Root Cause:** The `projects_v2` module uses `count = 1`, requiring resources to be accessed as `module.projects_v2[0]` instead of `module.projects_v2`. The target paths generated by `get_test_mode_targets()` were missing the `[0]` index.

**Solution:**
- Updated `get_test_mode_targets()` function in `test/run_e2e_test.sh` to include `[0]` index in target paths
- Changed target format from `module.dbt_cloud.module.projects_v2.dbtcloud_global_connection.connections["key"]` 
- To: `module.dbt_cloud.module.projects_v2[0].dbtcloud_global_connection.connections["key"]`

**Impact:**
- Test mode now correctly targets and plans/applies 1-2 resources
- Terraform recognizes the target addresses and executes plans/applies successfully
- Test mode workflow is now fully functional

## 🔧 Technical Details

### Connection Configuration Priority

The Terraform module now uses the following priority order for connection configuration:

1. **`provider_config`** (user-provided, from `.env` files) - **Highest Priority**
2. **`details`** (API-fetched data)
3. **Defaults** (test values) - **Lowest Priority**

**Implementation:**
```hcl
snowflake = try(each.value.type, "") == "snowflake" ? {
  account   = try(each.value.provider_config.account, try(each.value.details.account, "test-account"))
  database  = try(each.value.provider_config.database, try(each.value.details.database, "TEST_DB"))
  warehouse = try(each.value.provider_config.warehouse, try(each.value.details.warehouse, "TEST_WH"))
  # ... additional fields
} : null
```

### Test Mode Targeting Logic

The `get_test_mode_targets()` function:
1. Parses YAML to extract connection keys from `globals.connections`
2. Falls back to group keys from `globals.groups` if no connections found
3. Constructs Terraform target addresses with correct module path including `[0]` index
4. Returns space-separated list of targets for use with `-target=` flags

## 🚀 Upgrade Instructions

### For Users

**No breaking changes.** This release enhances test mode functionality without affecting production workflows.

**To use new features:**
1. Ensure connection credentials are defined in `.env` files (they will be automatically used in test mode)
2. Use `--test-destroy` flag to clean up test resources after testing

### For Developers

**Test Mode Workflow:**
```bash
# 1. Plan test resources
./run_e2e_test.sh --test-plan

# 2. Apply test resources (uses real data from .env)
./run_e2e_test.sh --test-apply

# 3. Clean up test resources
./run_e2e_test.sh --test-destroy
```

**Connection Configuration:**
- Connection data in `.env` files (via `DBT_CONNECTION_*` variables) is automatically injected into YAML
- Terraform module prioritizes `provider_config` over `details`
- Ensure `.env` files contain correct connection credentials for accurate testing
